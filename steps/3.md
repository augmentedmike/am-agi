# Step 3: The Runner

The runner is responsible for executing Wiggum loops. It contains the baked-in intelligence — the specific workflows, procedures, and system prompts that govern how work gets done. The loop itself is generic; the runner gives it meaning.

## What the Runner Does

The runner manages a Kanban board. Each card on the board is a unit of work. The runner picks the highest-priority card, runs a Wiggum loop iteration against it, and enforces the gating rules that govern when a card can move between columns.

```
runner → pick card → run iteration → check gates → move or stay
```

## The Board

The Kanban board has four columns. Cards flow left to right. Movement is gated — a card cannot advance until its exit criteria are met.

```
backlog → in-progress → in-review → shipped
```

The runner enforces these gates. The agent inside the loop does not decide when to move a card — the runner does, based on deterministic criteria.

## Columns and System Prompts

Each column has a system prompt that tells the agent what kind of work to do for cards in that state. The agent's behavior changes depending on which column the card is in.

### backlog

> Research and prepare. Read `work.md`. Explore the codebase, gather context, fetch relevant docs. Write `criteria.md` (acceptance criteria), `research.md` (cached sources), and `todo.md` (step checklist). Do not write code yet.

**Gate to in-progress:** `criteria.md`, `research.md`, and `todo.md` all exist and are non-empty.

---

### in-progress

> Implement. Read `work.md`, `criteria.md`, `research.md`, and `todo.md`. Do one meaningful unit of implementation work. Update `todo.md`. Write `iter/<n>/agent.log`.

**Gate to in-review:** All items in `criteria.md` have a corresponding implementation. Agent marks todo complete.

---

### in-review

> Verify. Read `criteria.md`. Run tests. Check each acceptance criterion. Confirm nothing regressed. If everything passes, signal ready to ship. If anything fails, log the failure and signal to return to in-progress.

**Gate to shipped:** All criteria verified, tests pass.
**Gate back to in-progress:** Any criterion fails — logged, work resumes.

---

### shipped

> Trigger the post hook. Squash all iteration commits to one clean commit, rebase onto main, merge, push, clean up the worktree and branch.

**Terminal state.** Cards do not leave shipped.

## Card Priority

Within any column, the runner works highest priority first:

| Priority | Meaning |
|---|---|
| critical | Blocks other cards or a hard deadline. Do first. |
| high | Important. Do before normal. |
| normal | Default. |
| low | Do when nothing else is waiting. |

## Runner Intelligence

The runner is not a generic task executor. It knows:

- Which column a card is in and therefore which system prompt to use
- What the gate conditions are for each transition
- How to read gate state from the filesystem (does `criteria.md` exist? do tests pass?)
- How to move a card via the `board` CLI when a gate is met
- How to log failures and reroute cards back to in-progress

This intelligence is baked into the runner, not delegated to the model. The model does the work inside each iteration. The runner decides what happens next.

## Prompt

```
Read docs/AGENT-LOOP.MD, docs/KANBAN.MD, steps/3.md, and CLAUDE.md. Build the runner at `src/runner/index.ts` using Bun and TypeScript.

The runner is a CLI entry point: `bun run src/runner/index.ts`. It runs one full Wiggum iteration against the highest-priority actionable card and exits.

Implementation:

1. PICK CARD
   - Call `board search --state in-progress` — take the highest-priority result
   - If none, call `board search --state backlog` — take the highest-priority result
   - If none, print "no actionable cards" and exit 0
   - Priority order: critical > high > normal > low

2. RESOLVE COLUMN → SYSTEM PROMPT
   - backlog: "Research and prepare. Read work.md. Explore the codebase, gather context, fetch relevant docs. Write criteria.md, research.md, and todo.md. Do not write code yet."
   - in-progress: "Implement. Read work.md, criteria.md, research.md, and todo.md. Write one meaningful unit of implementation with tests written first (TDD). Update todo.md. Write iter/<n>/agent.log."
   - in-review: "Verify. Check each criterion in criteria.md. Run tests. Scan for security issues. Confirm intent matches implementation. Check linting and code quality. Log pass or fail for each criterion. Write iter/<n>/agent.log. Do not move the card."
   - shipped: not reachable — runner never picks shipped cards

3. RUN ITERATION — call runIteration(cardId, workDir) from src/loop/index.ts

4. CHECK GATES — after the iteration commits, run the gate check for the card's current state:
   - backlog gate: criteria.md, research.md, and todo.md all exist and are non-empty → call `board move <id> in-progress`
   - in-progress gate: todo.md has no unchecked items → call `board move <id> in-review`
   - in-review gate: run verifier (see below) → pass: `board move <id> shipped`, fail: `board move <id> in-progress` with failure logged

5. VERIFIER (in-review gate check)
   - Run `bun test` — must exit 0
   - Parse iter/<n>/agent.log for explicit pass/fail per criterion
   - All criteria in criteria.md must have a corresponding pass entry
   - If any fail: call `board update <id> --log "<failure summary>"` before routing back

Code quality:
- Runner is a pure orchestrator — no business logic inline, only calls to named functions
- System prompts are constants in a dedicated file `src/runner/prompts.ts`
- Gate logic is pure functions in `src/runner/gates.ts`, independently testable
- Verifier is its own module `src/runner/verifier.ts`
- No process.exit() except at the top-level entry point
- All board CLI calls go through a typed wrapper in `src/cli/board.ts` — never raw shell strings scattered through the code

Definition of done:
- `bun test` passes for: card selection priority logic, gate conditions for each state, verifier pass/fail parsing
- Running the runner against a board with one backlog card completes one iteration, produces memory files, and moves the card to in-progress if gates are met
- A card in in-review with failing tests stays in in-review and has a failure entry in its work log
```
