# Step 5: The Kanban Board — A Prefrontal Cortex

The Kanban board is the agent's prefrontal cortex. It governs procedural, deterministic behavior — encoding 20+ years of project management, scrum, and agile practice into the system rather than leaving it to the model to reason about on the fly.

## Two Jobs

**1. Deterministic behavior.** The board enforces the procedures we want. Gate conditions, state transitions, work sequencing, priority ordering — none of this is up to the model. It is baked into the board and the runner that operates it. The agent cannot skip a phase, hallucinate a completed review, or ship unverified work.

**2. Mechanistic interpretability.** The board is a window into the agent's brain. Every card shows exactly where work stands, what was done, what failed, and what comes next. Nothing is hidden inside a model's context window. Everything is inspectable, auditable, and human-readable at any time.

## Why This Works

Agile and scrum are fundamentally state machines with gated transitions. Backlog grooming, sprint planning, in-progress limits, review gates, definition of done — these are the same concepts, formalized for agents. The board enforces them mechanically so the agent inherits decades of hard-won process discipline without having to reason about it.

| Practice | How It Maps |
|---|---|
| Definition of Done | `criteria.md` — written before work starts, verified before shipping |
| Acceptance criteria | Gate conditions on each column transition |
| Sprint review | The in-review column — explicit verification phase, not implicit |
| Work in progress limits | Runner picks one card at a time per priority |
| Backlog grooming | The backlog column — research and prep before implementation begins |
| Retrospective | `iter/<n>/agent.log` + work log — full record of what happened and why |

## Interpretability

Because all state lives in files and all transitions are logged, you can inspect the agent's reasoning at any point:

- **Where is the work?** Check the board column.
- **What is the agent doing right now?** Read `todo.md`.
- **What has it decided?** Read `criteria.md`.
- **What did it find?** Read `research.md`.
- **What happened in each iteration?** Read `iter/<n>/agent.log`.
- **Why did it move states?** Read the card's work log.

No black box. No opaque model state. The entire execution history is in git, readable by any human or tool.

## Quality Guarantee

The board structure guarantees quality by construction:

1. Work cannot start without defined acceptance criteria.
2. Implementation cannot be reviewed without meeting all criteria.
3. Work cannot ship without passing review.
4. Failures are logged and route the card back — nothing is silently ignored.

The agent does not grade its own work in isolation. The review phase is structurally separate from implementation. The runner enforces the separation. This is the same pattern that makes human code review effective — the reviewer and the implementer operate under different constraints, at different times.

## Tech Stack

The board is a full-stack Next.js application.

| Layer | Technology |
|---|---|
| Frontend | React 19+, Tailwind 4 |
| Backend | Next.js (API routes, WebSockets) |
| Database | SQLite + vector DB extension, Drizzle ORM |

**Why this stack:** Next.js gives us WebSockets for live board updates and a typed API to route through. React handles the UI. SQLite keeps the data layer simple and local — no external services required.

### Data Layer

All user data that is not a document, repo, or other filesystem artifact is stored here:

- **Board state** — cards, columns, priorities, work logs, timestamps
- **Agent metadata** — iteration records, gate transition history
- **Knowledge base** — vector embeddings for long-term memory; the agent can store and retrieve learned context across tasks without relying on model context windows

Documents, repos, and other artifacts live on the filesystem and in git. Everything else lives in SQLite.
