# Step 8: Column Gates and Column Prompts

Each column is a contract. The contract specifies what work must be done, what artifacts must exist, and what verification must pass before a card can leave. This step makes those contracts explicit in code: richer gate checks and column prompts that tell the agent exactly what to do when dispatched to a card.

## Column Prompts

Each column has a **column prompt** — a fixed, opinionated description of what "doing the work for this column" means. When AM is dispatched for a card, the column prompt is the work description.

Column prompts live in `apps/board/src/worker/prompts.ts` as exported constants.

## What Each Column Means

### Backlog — Research and Planning

The backlog column is where work becomes concrete. Leaving backlog requires answering: *what exactly needs to be done, and what do we need to know before starting?*

**For code tasks:** AM reads the relevant codebase, identifies the specific files and line numbers that need to change, and writes `research.md`. Without grounded research, the agent working in-progress is flying blind.

**For non-code tasks:** AM performs web or literature search, identifies relevant sources, and writes `research.md` with references and key findings.

In both cases, AM writes `criteria.md` — a numbered list of acceptance criteria. These are the contract. Everything downstream verifies against them.

**Gate: backlog → in-progress**
- `criteria.md` attached, non-empty, contains numbered acceptance criteria (`1. ...`)
- `research.md` attached, non-empty
  - Code tasks: must contain at least one file path reference (`src/...`)
  - Non-code tasks: must contain at least one URL or citation
- Card title is non-empty

### In Progress — Do the Work

AM has criteria and research. It executes.

**For code tasks:** AM edits files, writes implementations, and writes tests. Tests are not optional. AM maintains `todo.md` as a flat checklist. When the work is complete, it calls `board move <id> in-review`. The gate checks `todo.md` and runs `bun test`.

**Gate: in-progress → in-review**
- `todo.md` has no unchecked items (`- [ ]`)
- If the worktree contains any `*.test.ts` / `*.test.tsx` / `*.spec.ts` files: `bun test` exits 0

### In Review — Verify, Not Redo

AM's job here is adversarial: assume the implementation is wrong and try to prove it.

**For code tasks:** AM checks each criterion independently, runs the full test suite, and scans against `docs/CODE_QUALITY.md`. If anything fails, AM calls `board move <id> in-progress` with a failure log. If everything passes, AM writes `✓ <criterion>` for each criterion in `iter/<n>/agent.log` and calls `board move <id> shipped`.

**Gate: in-review → shipped**
- Latest `iter/<n>/agent.log` exists with `✓` for every criterion in `criteria.md`
- `bun test` exits 0
- No violations of `docs/CODE_QUALITY.md` rules in `git diff main...HEAD` (if that file exists)

**Gate: in-review → in-progress** — always allowed.

## Prompt

```
Read steps/8.md, CLAUDE.md, and apps/board/src/worker/gates.ts.

This step adds column prompts and enhances the gate logic.

**Part 1 — Column Prompts**

Create `apps/board/src/worker/prompts.ts` with three exported string constants:

`BACKLOG_PROMPT` — instructs AM to:
- Read the card title and any existing work log
- Determine if this is a code task or non-code task
- For code tasks: read relevant source files, identify specific files and line numbers that need to change, write `research.md`
- For non-code tasks: use web search to gather relevant sources, write `research.md` with references and summaries
- Write `criteria.md` with numbered acceptance criteria (each is a testable, binary outcome)
- Write `todo.md` as a flat checklist of steps derived from criteria.md
- Attach all files to the card: `board update <id> --attach research.md --attach criteria.md --attach todo.md`
- Attempt `board move <id> in-progress` — if rejected, address the failure and retry

`IN_PROGRESS_PROMPT` — instructs AM to:
- Read criteria.md, research.md, and todo.md
- Implement each unchecked criterion, checking it off when done
- For code tasks: write or update tests covering each criterion; run `bun test` before marking any criterion complete
- Rewrite todo.md with updated checkboxes after each criterion is completed
- When all criteria are checked off and tests pass: `board move <id> in-review`
- If the gate rejects: address the specific failure, then retry

`IN_REVIEW_PROMPT` — instructs AM to:
- Read criteria.md and the latest iter/<n>/agent.log
- For each criterion: independently verify it is met by the implementation
- For code tasks: run `bun test`; scan implementation against docs/CODE_QUALITY.md if it exists
- Write iter/<n+1>/agent.log with `✓ <criterion>` for each passing criterion, or `✗ <criterion>: <reason>` for failures
- If all pass: `board move <id> shipped`
- If any fail: `board move <id> in-progress` with `--log "<summary of failures>"`

**Part 2 — Enhanced Gates**

Update `apps/board/src/worker/gates.ts`:

backlog → in-progress:
- `criteria.md` attached, exists, non-empty, contains at least one line matching `/^\d+\./`
- `research.md` attached, exists, non-empty
  - If content contains a path pattern like `src/` — treat as code research, require at least one file path
  - Otherwise — treat as non-code research, require at least one URL (`http`) or citation
- Card title is non-empty (not just whitespace)

in-progress → in-review:
- `todo.md` attached, exists, has no unchecked items (`- [ ]`)
- If workDir is set and contains any `*.test.ts`, `*.test.tsx`, or `*.spec.ts` files: run `bun test` — must exit 0

in-review → shipped (keep existing + add):
- Keep: iter/<n>/agent.log exists, criteria verified via ✓ pattern, bun test passes
- Add: if `docs/CODE_QUALITY.md` exists, verify none of its "never do" rules appear in `git diff main...HEAD`

**Definition of done:**
- `bun test` passes for all gate conditions including the new research.md checks
- Column prompts are exported from prompts.ts
- Gate correctly identifies code vs non-code research
- In-review gate checks docs/CODE_QUALITY.md if present
- Existing tests continue to pass

To execute this step:
`am "$(cat steps/8.md)"`
```
