# Step 6: The CLI Layer

All agent work falls into one of two types:

- **Type A — Real work:** Coding, writing, researching, debugging, data analysis. The agent operates freely — editing files, querying databases, configuring systems, running commands. This is where the model's capability shines and flexibility is an asset.
- **Type B — Process work:** Quality checks, state transitions, workflow operations, anything that touches shared infrastructure or crosses system boundaries. The agent is forced to use a CLI tool. No exceptions.

## Why the Distinction Matters

Type A work is inherently local and reversible. A bad code edit can be diffed and rolled back. A poorly written paragraph can be rewritten. The cost of a mistake is low.

Type B work is not. Sending an email, moving a card, merging a branch, invoking an external API — these actions have side effects outside the agent's worktree. They may be irreversible. They touch shared state. They can leak credentials or trigger unintended consequences if the model is left to figure out the mechanics on its own.

Forcing Type B through CLI tools eliminates up to 98% of hallucination in this category. The model doesn't reason about how to send an email — it calls `notify send`. The tool knows how. The model supplies the arguments.

## How It Works

Instead of asking the agent to figure out a process action end-to-end:

> "Send an email to the team saying the build failed"

The agent calls a CLI command that encodes the entire operation deterministically:

```sh
notify send --to team --subject "Build failed" --body "$(cat iter/<n>/agent.log)"
```

The tool handles:
- Credential lookup (no model access to raw secrets)
- Request formatting and validation
- Rate limiting and retry logic
- Gating (is the agent allowed to send this? under what conditions?)
- Logging the action to the work record

The model cannot hallucinate the mechanics because it never sees them. It only sees the interface.

## Built-In Gating and Security

Each CLI tool encodes its own security model:

- **Credentials** are resolved by the tool, not passed by the model
- **Scope** is enforced — a tool that posts a comment cannot delete a repository
- **Gates** are checked before execution — a card cannot be moved to shipped if tests haven't passed
- **Audit trail** — every CLI invocation is logged with the arguments supplied and the result returned

The model suggests the action. The tool decides if it is allowed. This is the same principle as capability-based security — the agent has access to tools, not to the underlying systems those tools operate on.

## Examples

| Action | Type | How |
|---|---|---|
| Write a function | A | Agent edits files directly |
| Query a database | A | Agent runs SQL directly |
| Move a card | B | `board move <id> <state>` |
| Merge to main | B | Ship script (deterministic, no model involvement) |
| Send a notification | B | `notify send ...` |
| Create a new workspace | B | `new-next <name>` |
| Archive a card | B | `board archive <id> --reason "..."` |

## The Board Tool

The `board` CLI is the agent's interface to the Kanban board. It is the primary Type B tool — every interaction with board state goes through it.

```sh
board create --title "Add scoring logic" --priority high
board move <id> in-review
board show <id>
board search --state in-progress
board update <id> --log "tests passing, moving to review"
board archive <id> --reason "superseded by task-012"
```

The agent can request a card move. It cannot perform one. That distinction is load-bearing.

### The Board Is a State Machine with a Gatekeeper

When the agent calls `board move <id> shipped`, it is filing a request — not issuing an instruction. A deterministic worker process intercepts the request and runs its own verification before the move is allowed:

1. Does `criteria.md` exist and list acceptance criteria?
2. Does `research.md` exist?
3. Does `todo.md` show all steps completed?
4. Do the tests pass?
5. Does `iter/<n>/agent.log` exist for the current iteration?
6. Are all acceptance criteria verifiably met by the implementation?

If every check passes, the worker moves the card. If any check fails, the move is rejected — the worker logs the specific failure to the card's work log and returns an error to the agent. The card stays where it is.

The model never grades its own work. The worker does. The board enforces the result.

### Why This Matters

An agent asked to self-report completion will report completion. It is not lying — it genuinely believes the work is done. But belief is not verification. The worker has no beliefs. It checks the filesystem, runs the tests, reads the logs, and returns a binary result.

This is the same reason we don't let developers merge their own PRs without review. The separation between "I think this is done" and "this is verified done" is where quality lives.

## The Principle

Codify process actions into deterministic tools. The model supplies intent and arguments. The tool supplies correctness, security, and repeatability. Never ask a model to figure out how to do something that can be encoded once and reused forever.
