# Step 2: Build the Agent Loop

The agent loop is a simple "Ralph Wiggum" loop that uses the filesystem for persistence instead of relying on models to maintain their own coherence.

## How It Works

Each iteration is one-shot. The model does exactly one unit of work, writes its state to files, commits, and exits. The next iteration picks up where the last one left off — not from memory, but from the filesystem.

## Starting Point: work.md

The first iteration begins with a `work.md` file — a description of the work requested. Write it as detailed as possible. You get what you ask for.

`work.md` is the source of truth. The loop reads it. It is never modified.

## The Loop

```
[pre] → work.md → criteria.md → todo.md → [work] → commit → update todo.md → [post on ship]
```

1. **Read `work.md`** — understand what needs to be built
2. **Generate `criteria.md`** (first run only) — define what "done" looks like
3. **Generate `research.md`** (first run only) — gather and cache all relevant sources
4. **Generate `todo.md`** (first run only) — flat checklist of concrete steps
5. **Do the work** — one meaningful unit per iteration
6. **Write `iter/<n>/agent.log`** — record what happened
7. **Rewrite `todo.md`** — check off completed steps, note what's next
8. **Commit everything** — `<task-slug>/iter-<n>: <summary>`
9. **Exit** — next iteration picks up from the files

## Memory Files

The first iteration produces three memory files. These are the agent's persistent context — read at the start of every subsequent iteration.

| File | Purpose |
|---|---|
| `work.md` | What to build (input, never modified) |
| `criteria.md` | Acceptance criteria — what "done" looks like |
| `research.md` | Cached research — code, web, papers, or other sources gathered upfront |
| `todo.md` | Flat step checklist — read at start, rewritten at end of each iteration |
| `iter/<n>/agent.log` | What happened in iteration n |

### research.md

The first iteration does all its research and writes the findings to `research.md`. This precaches everything the agent will need — relevant source files, API docs, prior art, web results, papers — so subsequent iterations don't re-fetch or re-explore. The model reads `research.md` instead of going back to the source.

No model memory required. Each iteration bootstraps entirely from the filesystem.

## Subsequent Iterations

In iterations after the first, all previous files are available to the agent — but not all are loaded into context automatically. Only what is needed for the current step is loaded. This keeps context windows lean and prevents the model from being distracted by irrelevant history.

## Implementation and TDD

Once research, criteria, and todo exist, subsequent iterations move into implementation. The agent writes code against the acceptance criteria in `criteria.md`, guided by the steps in `todo.md` and the sources cached in `research.md`.

Implementation includes tests — written first where possible. Test-driven development (TDD) is a first-class requirement, not a suggestion. Writing the test before the implementation is one of the most effective mechanisms for reducing slop in both human and AI-generated code. A test that is written to pass existing code proves nothing. A test written first defines intent, and implementation that satisfies it is demonstrably correct.

## Verification

At some point the iterations advance the card into the verification stage. This is a structurally separate phase — the agent is no longer implementing, it is auditing. Verification checks:

- **Tests** — all tests pass, coverage is adequate
- **Refactors** — code is clean, no obvious structural debt
- **Security** — known vulnerability patterns are scanned and flagged
- **Intent alignment** — the implementation matches the description in `work.md` and all criteria in `criteria.md`
- **Code quality** — linted, formatted, scores well on known good-software metrics

If all checks align, the agent writes its files and exits — the board worker will verify independently and move the card to shipped.

If any check fails, the agent writes its files and exits — logging the specific failures so the next iteration can address them. The card stays in review. The board worker will confirm the failure and route the card back to in-progress.

In either case, the agent writes and exits. It does not retry, loop internally, or attempt to move the card itself. Board movement is locked in the state machine. The agent is explicitly instructed not to hack it.

## Prompt

```
Read docs/AGENT-LOOP.MD, steps/2.md, and CLAUDE.md in this repo. Build the Wiggum loop as a TypeScript module at `agent/src/loop/index.ts` using Bun as the runtime.

The loop is a single exported function `runIteration(workDir: string)` that does exactly this:

1. READ — load work.md (required), and whichever of criteria.md / todo.md exist
2. INVOKE CLAUDE — run `claude --dangerously-skip-permissions -p <prompt> --output-format json` with `cwd` set to `workDir` so Claude operates inside the worktree, and pass its stdout through
3. EXIT

When the implementation is complete and tests pass, commit ALL files as the very last action — after every file has been written: `git add -A && git commit -m "step 2: build Wiggum loop"`. Verify `git status` is clean after the commit.

Code quality requirements:
- Each step is a clearly named function — no god functions
- All file I/O goes through a FileSystem adapter interface (enables tests without touching disk)
- Errors name the step and path that failed
- Types for data shapes
- Tests written before implementation (TDD) using Bun's built-in test runner

Definition of done:
- `bun test` passes
- No test mocks the filesystem — use a real temp directory
```
